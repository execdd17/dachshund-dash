# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Dachshund Dash is a pixel-art endless runner (Chrome Dino-style) starring a dachshund. It's a static site built from native ES modules: `index.html` is a thin HTML shell, `css/style.css` holds all styles, and the game lives in `js/` as vanilla-JS modules rendered onto a `<canvas>`. There is no build step, bundler, or framework; `package.json` exists only to run the unit tests.

Live at https://execdd17.github.io/dachshund-dash/ (deployed via GitHub Pages from this repo).

## Common Commands

```bash
# Run locally — must be served over HTTP (ES modules don't load from file://)
python3 -m http.server 8000      # then visit http://localhost:8000

# Unit tests (Node 18+ built-in test runner, zero dependencies)
npm test                         # or: node --test tests/*.test.js

# Firestore rules + indexes (Firebase CLI; see firebase/README.md)
npm run firebase:deploy
```

There is no lint or build tooling. Verify gameplay changes by running the tests and playing the game in a browser (check the DevTools console for errors, especially around asset loading and Firebase init). For headless verification, `.claude/skills/verify/SKILL.md` documents a working playwright-core recipe (driving the game, reading canvas state, DOM signals).

## Architecture

The code is layered so that game logic never touches the DOM, canvas, or Web Audio directly — everything effectful is injected. That's what makes the `tests/` suite possible: all of `core/`, `systems/`, and `leaderboard/` runs under plain Node.

### Layers (js/)

- **`config.js`** — every tuning constant (physics, speeds, spawn chances, encounter timings, sprite scales, storage keys). Pure data; imported everywhere.
- **`core/`** — `state.js` (the single mutable state object: `createState()`, `resetRun()`; field names match the pre-refactor globals), `timeOfDay.js` (score-driven day/night cycle), `color.js` (lerpColor).
- **`systems/`** — pure-ish gameplay logic operating on `(state, …, services)`:
  - `update.js` — per-frame orchestrator (physics, scrolling, scoring, weather) that calls the others
  - `control.js` (jump/duck + run reset), `spawning.js`, `collision.js` (hitbox math + resolution, incl. heart spend/i-frames), `giant.js`, `chase.js`, `boss.js`, `trampoline.js`, `encounters.js` (shared `*Busy()` mutual-exclusion predicates), `death.js`, `weather.js`
  - `services` = `{ sfx, music, globalScores, recordScore }`; randomness and clock are injectable (`rng`, `now`) for deterministic tests.
- **`render/`** — canvas drawing only, all functions take `(ctx, state, …)`: `draw.js` (frame orchestrator), `view.js` (world→canvas mapping: full-viewport sizing on every device, extended sky, safe-area insets), `background.js`, `obstacles.js` (incl. the per-type/skin dispatch in `drawObstacle`), `actors.js` (dog/squirrel sprites + cosmetic overlays), `effects.js`, `hud.js`, `primitives.js`.
- **`audio/`** — `sfx.js` (Web Audio-synthesized effects behind a `createSfx()` interface; `createSilentSfx()` for tests) and `music.js` (normal + giant-mode EDM tracks, autoplay unlock, `createSilentMusic()` for tests).
- **`assets/sprites.js`** — PNG sprite loading and frame-advance state for dog/squirrel/bird; the frame-selection helpers are pure.
- **`cosmetics/`** — `cosmetics.js` (slot/item registry `COSMETIC_DEFS`, head-anchor calibration data, equipped-state persistence) and `menu.js` (DOM overlay UI; the panel caps itself to the screen height with only the item grid scrolling, and on screens under 480px tall it goes wide — compact preview, ~8 tiles per row — so the DONE button stays reachable in phone landscape and the grid only scrolls once a slot has many items).
- **`leaderboard/`** — `local.js` (localStorage top-5, storage injected), `checksum.js`, `global.js` (Firestore fetch/submit + pure filtering), `ui.js` (HTML table).
- **`input/input.js`** — keyboard/touch/mouse wiring, including debug keys (P slow-mo, C chase, B boss, T trampoline scene, G golden hot dog, R rain, H hat).
- **`ui/nameEntry.js`** — high-score name overlay.
- **`main.js`** — composition root: the only module that imports everything; builds services, wires DOM controls, runs `requestAnimationFrame` loop calling `update()` then `draw()`.

### Key runtime facts

- `state.gameState` is the state machine: `setup` (start overlay: name + difficulty, once per page load) → `idle` → `running` → `dead` → back to `running`. Qualifying scores are recorded on death under the setup name (`services.recordScore`) — there is no post-death name prompt.
- Logical world is `W=800, H=250` (config.js); all coordinates are in logical space. The canvas fills the viewport on every device: `render/view.js` fits the world to the width (ground pinned to the bottom, extra height becomes sky above y=0 — `extraTop`; systems see it as `state.skyTop ≤ 0`) and recomputes `view.scale` on resize, so canvas-pixel positions depend on the window size (use `view.scale`/`view.extraTop`, never a hardcoded scale).
- The game was tuned at a 120Hz-equivalent baseline; `update()` scales by `dt / DT_BASELINE` for frame-rate independence, and dt is capped at 100ms.
- The dachshund/squirrel are PNG sprite sequences (`png/…`); the bird obstacle is too (`png/bird/bird_fly_*.png`, 6 frames derived from `png/bird/bird_spritesheet.jpg` by `tools/gen-bird-frames.sh`; `drawBirdSprite` in `render/obstacles.js` draws it `BIRD_SPRITE_W` wide with the talon baseline pinned just under the hitbox bottom, falling back to the procedural `drawBird` while frames load). Every other obstacle is drawn with canvas primitives. Chase mode re-skins birds as acorns before the sprite path is reached, so it's unaffected.
- Encounters (squirrel chase, boss, giant mode, trampoline scene) are strictly mutually exclusive — no overlap whatsoever, including transition phases. `systems/encounters.js` centralizes the busy predicates (`giantBusy`/`chaseBusy`/`bossBusy`/`trampBusy` + `goldenOnField`); every start check uses them, so a new encounter (or new phase) registers its flags there once. On top of exclusion, the three *scenes* (chase, boss, trampoline — not giant mode) guarantee `SCENE_MIN_GAP` ms (config.js, 15s) of normal gameplay between one another: each stamps `state.lastSceneEndAt` via `markSceneEnd` when its final transition phase finishes.
- **Scene queue (fairness)**: scenes take turns via a FIFO queue (`state.sceneQueue`, helpers in `encounters.js`). When a scene's own score condition is met (`CHASE_COOLDOWN`, `TRAMP_COOLDOWN`, boss milestones) it calls `requestScene(state, id)` — at most one queued entry per scene, so requests never stack. It then only goes pending when `tryStartScene(state, id, now)` grants the turn: it must be at the queue head AND the shared field rules must allow it (no encounter busy, no golden on the field, `sceneGapElapsed`). This exists because scenes used to arm independently with `update()`'s fixed call order (chase → boss → tramp) deciding ties; chase's short cooldown made it re-eligible at nearly every unblock moment, so it won every tie and the trampoline scene could starve for a whole run. A boss milestone crossed while blocked just sits in the queue — it is never lost. A pending scene stands down if giant mode starts anyway (giant keeps its normal edible field) but keeps its turn via `requeueSceneFront`, re-arming right after the shrink. A new scene must follow the same request/tryStart pattern and register its busy flags in `encounters.js`.
- **Trampoline scene** (`systems/trampoline.js`): requests its queue turn at score `TRAMP_FIRST_AT` (then every `TRAMP_COOLDOWN`), and once granted waits for a clear field, then runs `TRAMP_BOUNCES` reps of a thorn patch with a trampoline island (`getTrampLayout` computes tile counts from `TRAMP_REPS` × current speed — layout scales with speed so the patch is never double-jump bypassable). Landing on the island auto-bounces (`TRAMP_BOUNCE_VY`, resets the double jump, +`TRAMP_BOUNCE_BONUS`); thorn contact is terrain damage — spends a heart via the normal i-frame path but does **not** splice the tile or bonk-fling it, and a rising dog (vy < 0, i.e. mid-jump or mid-bounce) passes over thorns unharmed. The trampoline obstacle itself is never harmful. Normal spawning, goldens, chase, and boss are all suppressed while pending/active (see the encounter exclusion rules above). The obstacle cull in `update.js` is right-edge-based so the wide trampoline doesn't vanish early.
- Music requires a user interaction to start (browser autoplay policy) — see `music.attachAutoplayUnlock()`.
- **Hearts (lives)**: runs start with `state.startingHearts`, set by the difficulty slider on the start overlay (`DIFFICULTY_LEVELS` in config.js: VERY EASY 6 → VERY HARD 1, default NORMAL 3; locked per session — reload to change). A non-fatal hit in `checkCollision` spends a heart, knocks the obstacle away with the giant-mode bonk effect, and grants `HEART_HIT_INVULN` ms of i-frames (`state.invulnUntil`; the dog flickers in `draw.js`). A hit on the last heart kills. The HUD hearts (`drawHearts` in `render/hud.js`, pixel-art bitmap) sit top-left below the speaker/giant-timer slot; boss catches bypass hearts and kill directly. When giant mode ends, `deactivateGiantMode` grants `GIANT_END_INVULN` ms of the same i-frames, so an obstacle overlapping the dog at the shrink moment can't kill.
- **Layout** (full-bleed everywhere): `main.js` puts `app-mode` on `<body>` unconditionally; the CSS then splits on `touch-device`. Touch devices lock to the viewport — landscape is game-only (page furniture hidden), portrait hides the game and shows a rotate prompt plus the browsable leaderboard. Desktop (no touch — a laptop can't rotate) shows the full-viewport game at the top of a normally scrollable page with hints + global leaderboard below the fold; the setup-state keydown handler in `input/input.js` suppresses Space/arrow page scrolling under the overlay. The legacy `?app=1` query param is accepted but now a no-op. The game is also installable to a phone home screen (`manifest.json` + `icons/`), which just gets the same touch layout without browser chrome.

### Extending the game

- **New obstacle type**: add spawn logic in `systems/spawning.js`, a hitbox case in `systems/collision.js` (`getObstacleHitbox`), and a draw case in `render/obstacles.js` (`drawObstacle`). The trampoline scene's `'thorns'`/`'trampoline'` types follow this pattern but spawn from `systems/trampoline.js` (`spawnTrampRep`) instead of `spawning.js`, and get special resolution branches in `checkCollision` (terrain heart-spend / never-harmful).
- **Spawn patterns & bird walls** (`systems/spawning.js`): past `PATTERN_FIRST_AT`, each spawn cycle has a `PATTERN_CHANCE` roll to arm a multi-beat sequence (`SPAWN_PATTERNS`) with `PATTERN_CLOSE_GAP` between beats — e.g. duck under frisbees then jump a hot dog, or jump a hot dog then duck a bird stack. Past `BIRD_WALL_FIRST_AT`, solo spawns can also roll `birdStack` (two birds at `BIRD_HIGH_Y` + `BIRD_LOW_Y`, no vertical gap) or `birdTriangle` (low outer pair at `±BIRD_TRI_DX`, mid pair at `±BIRD_TRI_ROW2_DX` on `BIRD_WALL_MID_Y`, high pair on `BIRD_WALL_HIGH_Y`). Wall birds carry `aerialWall: true`. State lives in `state.activeSpawnPattern` and clears on golden spawn or the last beat.
- **Chase acorns**: during chase, solo `bird`/`frisbee` obstacles render as `ACORN_W×ACORN_H` acorns pinned to `CHASE_ACORN_Y` (duck-under). `aerialWall` low-row birds (`wallRow: 'low'`) use the same duck height for draw and hitbox; high-row birds keep `BIRD_HIGH_Y` with an acorn-sized hitbox. Solo-bird `BIRD_JUMP_BONUS` does not apply to `aerialWall` birds.
- **New cosmetic item**: two kinds. *Anchor items* (hats, sunglasses): drop the PNG under `png/<slot>/…` and add an entry to `COSMETIC_DEFS` in `cosmetics/cosmetics.js`; per-slot placement is `SLOT_ANCHOR_OFFSET` + the pose-calibrated `HEAD_ANCHORS`, with optional per-item `offset` overrides (scale/dx/dy) merged on top by `getItemAnchorOffset`. Five of the glasses (star/round/3d/shutter/monocle) are generated by `node tools/gen-glasses.cjs .` — 440x260 chunky-pixel PNGs built on a 10px cell grid; add a `gen*` function + `DESIGNS` entry there for a new pair (translucent lens colors use alpha). *Per-frame items* (the shoes in the clothes slot, everything in the body slot): one full-size transparent overlay PNG per dog frame plus the bite sheet (`item.frames` dir; `item.image` is just the menu icon), selected by `getOverlayFrame` and drawn with the base frame's exact geometry so they track the paws in every pose. The sneaker overlays are generated by `node tools/gen-shoes.cjs .` from hand-picked per-frame paw boxes — add a palette to `STYLES` there for a new colorway. The tutu overlays come from `node tools/gen-tutu.cjs . [previewDir]`, which paints a waistband + tulle frills around a hand-picked per-frame waist anchor (`SPECS` there), auto-fitting to the body silhouette. The three neck outfits (red bandana, hero cape, studded collar) come from `node tools/gen-outfits.cjs . [previewDir]` — distinct painters (kerchief drape / billowing cloth / studded band + tag) sharing one hand-picked per-frame *neck* anchor table (`SPECS`: cx/cy/angle, optional rTop/rBot overrides where the silhouette scan would ride up the skull, plus per-item sub-overrides like `cape.capeAngle`); band-type regions are clamped to the body silhouette, cloth may hang past it. A new neck item = a new classify function + `ITEMS` entry there, reusing the anchors. Both generators' `[previewDir]` emits per-frame composites for eyeballing alignment.
- **New tunable**: add it to `config.js`, never inline.
- **New test**: `tests/*.test.js`, Node built-in runner; use `tests/helpers.js` (fake storage, sequence rng, silent services). Keep browser-only imports (render/, input/, ui/, menu, main) out of tests.

### Leaderboards

- **Local**: top scores in localStorage (`leaderboard/local.js`, capped at `MAX_HIGH_SCORES`). Not rendered as a panel anymore — the only in-game surface is the `HI` figure top-right (`drawScore` in `render/hud.js`, best local score). The idle/game-over screens instead show the DOM ACCESSORIZE button (`#customizeBtn`, with a top-hat PNG icon), centered below the status text (CSS positions it at logical y≈118 via a `vw`-based offset from the wrapper bottom, matching the canvas's width-fit scale).
- **Global**: Firebase Firestore `scores` collection (`firebase.js` + `leaderboard/global.js`), **per difficulty**: every submission carries the session's difficulty label (`state.difficulty`, e.g. `'NORMAL'`), the UI (`leaderboard/ui.js`) shows one board per difficulty with tabs to switch. After setup the player lands on **their** difficulty tab (not a hardcoded NORMAL default) and the game fetches up to `GLOBAL_MAX_SCORES` (100) rows for that difficulty only; browsing another tab fetches that board on demand (cached client-side so switching back is free). The HTML table still paginates 10 rows at a time client-side. `qualifies(score, difficulty)` and game-over `placement(score, difficulty)` read the cached session board — no extra Firestore calls on death. Ties rank below existing entries; null placement means no banner. Submissions carry a simple non-cryptographic checksum (`checksum.js`, DJB2-style salted hash over name|score|timestamp|difficulty) as friction against casual tampering — not real security. **Composite index** in `firestore.indexes.json`: `difficulty` ASC, `score` DESC, `timestamp` ASC.
- The Firebase config/API key in `js/firebase.js` is a public client-side web config (expected to be visible in a static site) — don't treat it as a secret to redact, but also don't assume it's a security boundary. Firestore rules gate write access.
- **Firestore admin (local)**: `firebase/README.md` — rules/indexes deploy via Firebase CLI (`npm run firebase:deploy`); bulk data ops via Admin SDK scripts in `firebase/` (service account key at `firebase/service-account.json`, git-ignored). `firestore.rules` strictly validates the score doc shape — exactly the keys `name, score, difficulty, timestamp, checksum`, plus type/range checks (name 1–12 chars, difficulty one of the five labels, score 0–100000, timestamp near server time). **Any change to the submitted doc shape or these constraints requires updating `firestore.rules` and `firestore.indexes.json` if needed, then `npm run firebase:deploy`** — otherwise submissions fail with PERMISSION_DENIED (just a `console.warn` in-game). Bulk deletes: `firebase/delete-scores-by-difficulty.mjs`, `firebase/delete-scores-by-name.mjs` (default empty name → `PLAYER`).

### Input

- Desktop: keyboard (Space/Up jump, Down/S duck) and mouse click/tap.
- Mobile: touch-only, screen-position-based — right side jumps, holding the left `DUCK_ZONE_RATIO` (40%) ducks. Touch detection adds a `touch-device` body class used by the CSS for mobile hints.
